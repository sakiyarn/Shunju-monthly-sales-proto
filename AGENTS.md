# Project Working Rules

このリポジトリで作業する AI / 開発者は、以下の方針を必ず守ること。

## 1. 旧デモ画面の扱い
- 旧フロントデモ（Vue + Vite 単体構成）は、比較検証用として残す。
- マスタデータ管理のセクション2〜10の CRUD 実装と受け入れ確認が完了するまでは削除しない。
- 削除判断は、全セクションの移植完了後に実施する。

## 2. ブランチ運用
- 機能実装はセクション単位でブランチを分ける。
- ブランチ名は `codex/` プレフィックスを必須とし、以下を推奨する。
  - `codex/master-s2-projects-crud`
  - `codex/master-s3-assignments-crud`
- 1ブランチ 1目的を徹底し、レビューしやすい差分にする。

## 3. 開発フロー
- 既存ブランチで作業中の変更は、先にコミットしてから次の作業に入る。
- 可能であれば機能ごとに PR / マージを行い、段階的に `main` へ取り込む。
- 大きな一括変更より、小さな変更を積み上げる。

## 4. 実装方針（マスタデータ画面）
- セクション1で確立した Inertia CRUD パターンを、セクション2〜10にも順次適用する。
- 画面表示責務は `MasterController`、更新責務はリソース別 Controller に分離する。
- 既存 UI の見た目・操作感はできるだけ維持し、データ取得を props 化して置き換える。

## 5. スコープ運用（クロスチャット共通）
- マスタデータ画面を触る場合は、作業開始前に必ず `docs/master-migration-scope.md` を読む。
- S5で「S3/S5統合UI」を検証する場合は、`docs/master-s5-unified-ui-trial.md` も着手前に読む。
- タスク開始時に「今回の対象セクション番号（例: S3）」を明示してから着手する。
- スコープ外の変更は実装せず、必要があれば影響調査メモとして残す。
- 新しいチャットを開始する場合は `docs/templates/master-section-kickoff.md` をコピーして前提を共有する。
- PR作成時は `docs/templates/master-section-pr-checklist.md` を使い、非対象セクションを明示する。
