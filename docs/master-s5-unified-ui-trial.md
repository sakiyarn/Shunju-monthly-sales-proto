# S5統合UI 検証メモ（S3/S5議論ログ）

## 背景
- 現場運用はスプレッドシート中心で、1画面内で操作完結する体験が望まれている。
- 現在の設計は S3（案件メンバーアサイン）と S5（請求稼働）を分離している。
- 相談テーマは「S5だけで運用完結できるか」「どの進め方で比較検証するか」。

## 合意した方針
- UIは S5 で完結する方向で検証する。
- ただしデータ責務は分離を維持する（S3=マスター、S5=月次実績）。
- すなわち「UI統合 / データ分離」のハイブリッドを第一候補とする。

## S3とS5を分けるメリット（再確認）
- マスター情報と実績情報の責務分離が明確になる。
- 過去月データの保全がしやすい（当月変更が過去に波及しにくい）。
- 枠作成ロジックの起点（S3→S5）を定義しやすい。
- 権限分離や障害切り分けがしやすい。

## 要件メモ（統合UIで実現したい運用）
- 新しい月で「枠の作成」ボタンを押すと対象月列を生成し、先月単価を引き継ぐ。
- 必要時にメンバーの新規アサインを追加できる。
- メンバー離任時は対象月以降の稼働/単価を空にし、次月枠作成でも空のままにする。
- 案件終了時は編集不可化し、終了一覧（トグル領域）で閲覧・再開できる。

## 実装上の重要ポイント
- 離任要件を堅牢にするため、`project_members` に月次有効期間（例: `start_month`, `end_month`）導入を優先検討する。
- 枠作成APIは冪等にする（既存データを二重作成しない）。
- 単価引継ぎは「先月実績単価 > S3デフォルト単価」の優先順を基本ルールにする。

## 推奨ブランチ戦略（比較実験向け）
1. `main` から `codex/master-s5-worklogs-crud` を作成
2. S5の通常版（分離前提）を最小実装してベースラインを固定
3. そのブランチから `codex/master-s5-unified-ui-spike` を作成
4. 統合UI案を実装し、同一観点で比較
5. 微妙なら `codex/master-s5-unified-ui-spike` を破棄し、検証メモのみ残す

## 比較観点（同じ手順で評価する）
- 月初の枠作成の速さと誤操作率
- 先月単価引継ぎの妥当性
- 新規アサイン追加のしやすさ
- 離任者の対象月以降制御のわかりやすさ
- 案件終了/再開時の運用負荷
- 画面の見通し（スクロール量、視認性）
