# マスタ移植スコープ運用ガイド（S2〜S10）

## 目的
- 新しいチャットを立ち上げても、毎回同じ前提で実装判断できるようにする。
- 1ブランチ1目的を徹底し、レビュー容易性と変更の安全性を維持する。

## 全体原則
1. 1ブランチ1セクションを守る。
2. 対象セクション以外は実装しない。
3. スコープ外で気づいた課題は「影響メモ」に残し、次ブランチへ回す。
4. 既存の UI 操作感を維持しつつ、段階的に Inertia CRUD パターンへ移行する。

## スコープ定義テンプレ
以下をチャット冒頭で固定する。

```md
対象セクション: Sx
In Scope:
- （今回実装する内容）

Out of Scope:
- （今回実装しない内容）

影響確認のみ:
- （他セクションへの影響を調査する観点）
```

## セクション別の進め方（S2〜S10共通）
1. 対象セクションの現行挙動を確認する。
2. `MasterController` の表示責務と対象リソースControllerの更新責務を分離する。
3. props化したデータで画面を置換し、ダミーデータ依存を減らす。
4. 手動受け入れ観点を明文化して完了確認する。

## 他セクション影響の扱い
- 仕様化・メモ化は実施してよい。
- ただしコード実装は次の対象セクションブランチで行う。
- 例: S3実装中にS4影響を発見した場合、S3内ではメモのみ残す。

## セクション2で実際に出た影響検討メモ（再利用用）
- セクション3: 終了案件を表示するか、編集可否をどうするかを先に決める。
- セクション5: 終了案件を閲覧対象に残すかどうかを先に決める。
- セクション9/10: 終了案件を新規選択肢から除外する場合、既存参照値の保全方法を先に決める。
- 案件ステータス表現は `status(open/closed)` ではなく `is_active(boolean)` を基準に統一する。
- `client_name` は段階移行対象として扱い、削除タイミングを別タスクで管理する。
- 新しい並び順仕様（`display_order`）を導入した場合、他セクションの並び基準との整合を確認する。

## 影響確認の観点テンプレ（流用可）
```md
- [ ] 参照セクション（S3/S5/S9/S10）に新しい状態項目の前提ズレがない
- [ ] 既存データ（終了案件など）の表示・参照を壊していない
- [ ] 段階移行項目（例: client_name）の扱いを今回スコープで変更していない
- [ ] 表示順ルール（display_order など）が他画面と矛盾しない
```

## 受け入れ基準テンプレ
```md
- [ ] 対象セクションの主要CRUD操作が意図どおり動く
- [ ] Out of Scopeセクションに意図しないコード変更がない
- [ ] エラー時の表示（トースト/バリデーション）を確認した
- [ ] 手動確認観点をPR本文に記載した
```

## ブランチ命名とPR粒度
- ブランチ名は `codex/master-sX-<topic>` を使う。
- 1PRで扱う目的は1つに限定する。
- 既存未コミット変更がある場合は、着手前に整理方針を明示する。

## 既知の注意点
- 旧フロントデモ（Vue + Vite 単体構成）は比較検証用に残す。
- 全セクション移植完了まで削除しない。
- `client_name` など段階移行中の項目は、互換性を優先して扱う。

## 運用手順
1. 新チャット開始時に `docs/templates/master-section-kickoff.md` を貼る。
2. 実装後は `docs/templates/master-section-pr-checklist.md` でPR説明を作る。
